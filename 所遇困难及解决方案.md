# 所遇困难及解决方案

## 1. 预训练模型微调后生成的首字为 [UNK]

### 现象

微调完成后的 `uer/gpt2-chinese-poem` 模型在生成下句时，部分输入（约 1/5 的测试用例）的下句首字为 `[UNK]`（token id = 100），例如：

```
上句: 床前明月光  →  下句: [UNK]寒泉聲
```

而未经微调的原始预训练模型在该位置能生成有效汉字，说明微调过程引入了该问题。

### 排查过程

1. **排除位置编码问题**（推测阶段）：
   - 训练时使用 right-padding，`[SEP]` 在 position 6-8 附近
   - 首次尝试生成时使用了 left-padding，`[SEP]` 被推到 position ~60
   - GPT-2 使用绝对位置编码（learned），pos embedding 不一致会影响输出
   - **验证：** 改为无 padding 输入后问题依旧，排除此原因

2. **检查训练数据标签**：
   - 验证集 103,782 个样本中，**975 个（0.94%）的下句首字标签为 `[UNK]`**
   - 共 6,871 个样本（6.6%）的标签包含至少一个 `[UNK]` token
   - `[UNK]` 总占比 0.13%

### 根因

**数据预处理阶段未过滤 tokenizer 的 OOV 字符。**

导致这一问题的完整链路：

```
原始数据（含生僻字）
    ↓
自定义词表过滤（保留频次≥3的字符）
    ↓
tokenizer 编码（uer/gpt2-chinese-poem 的 BertTokenizer 词表中无此字）
    ↓
生僻字 → [UNK]
    ↓
labels 中包含 [UNK]
    ↓
训练时，CrossEntropyLoss 在 [UNK] 位置计算损失
    ↓
模型学会将 [UNK] 作为合法预测输出
```

`save_pretrained_processed_data()` 在生成训练数据时使用了自定义词表的频率过滤，但没有检查 BERT tokenizer 是否真正认识这些字符。古诗词中约有 0.94% 的诗句下句首字不在 `uer/gpt2-chinese-poem` 的词表内，导致模型学到了以 `[UNK]` 开头。

### 解决方案

修复 `utils/dataset.py` 中的 `save_pretrained_processed_data()` 函数，在生成训练数据前**用 tokenizer 正向过滤 OOV 样本**：

```python
# 对每对诗句，检查下句中是否有字符被 tokenizer 映射为 [UNK]
# 若有则跳过该样本
def is_valid_pair(upper, lower, tokenizer, max_len=64):
    """检查诗句对是否所有字符都在 tokenizer 词表中。"""
    # 构建完整序列并 tokenize
    upper_spaced = " ".join(list(upper))
    lower_spaced = " ".join(list(lower))
    full_text = f"[CLS] {upper_spaced} [SEP] {lower_spaced}"
    encoded = tokenizer(full_text, add_special_tokens=False)
    # 检查 [UNK] token (id=100)
    return 100 not in encoded["input_ids"]
```

**修改后需要重新运行：**
1. `python utils/dataset.py --mode pretrained` 重新生成过滤后的训练数据
2. `python train.py pretrained` 重新微调

### 影响

- 过滤后会丢弃约 6-7% 的训练样本（含 OOV 字符的诗句对）
- 不影响模型对常见字的生成效果，反而会提升生成质量
- 属于数据质量问题，不涉及模型架构改动
